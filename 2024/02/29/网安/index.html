渗透测试流程

1.确定目标

2.信息收集：
主动信息收集（有流量、数据传输，行为会被记录，无法隐蔽）：直接访问网站
被动信息收集：ip 信息、网段、子域名、开放端口、人员信息、操作系统版本、网站指纹

3.漏洞探测：
awvs 等

4.漏洞验证：
验证漏洞是否真实存在

5.漏洞利用：权限提升，权限维持，内网代理，内网收集，内网漏洞，横向移动，跨域攻击，痕迹清理

6.信息整理

7.攥写报告

子域名爆破时解决泛解析的方法

泛域名解析：为了使用户在输错域名时也能访问到我们的 ip 地址，将所有的子域名都解析到我们的 ip 地址上。

解决方法：首先访问一个不存在的域名来确定是否有泛域名解析的问题，后利用脚本不断生成不同的域名并发送请求，并记录下每次的 ip 和 ttl 直到大部分 ip 出现 2 次，ip 黑名单收集完毕。随后使用域名字典进行爆破，根据黑名单进行过滤，同时比较 ttl ，在泛解析记录中 ttl 是相同的。

如何绕过 CDN 查找真实 IP

CDN：内容分发网络，使用户就近获得所需内容，降低网络阻塞。

判断 CDN 的使用：多地 ping，查看 IP 地址是否唯一。

绕过：

1.国外代理访问

2.查询子域名 ip

3.phpinfo 文件

4.Mx 记录邮件服务。忘记密码、注册等功能会通过邮件，然后邮件原文中能够发现发送邮件的 ip 地址（邮件服务器需要搭建在自己的网站上面）

5.查询历史 DNS 记录。A 记录能够通过域名查找 IP 地址。https://dnsdb.io/zh-cn

DNS 记录

A 记录（主机记录）：一个主机域名和它对应的 IP 地址 www.baidu.com. IN A 1.1.1.2

NS 记录（域名服务器记录）：指定域名和解析它的 DNS 服务器 baidu.com. IN NS ns1.baidu.com.

SOA 记录（起始授权记录）：用于一个区域的开始，SOA 记录后的所有信息均是用于控制这个区域的，每个区域数据库文件都必须包含一个 SOA 记录，并且必须是其中的第一个资源记录，用以标识 DNS 服务器管理的起始位置，SOA 说明能解析这个区域的 dns 服务器中哪个是主服务器

CNAME 记录（别名记录）：域名和别名的记录。例如我们访问的百度网站的真实域名就是www.a.shifen.com，我们使用的www.baidu.com只是www.a.shifen.com的别名而已

MX 记录（邮件交换记录）：指定邮件服务器的域名

PRT 记录（逆向查询记录）：A 记录的逆向记录

phpinfo 中的信息

1.网站真实 IP，当网站使用 cdn 时有用。

2.网站绝对路径，直接写入 webshell 带到 getshell 的目的

3.allow_url_include 字段是否开启，开启了就支持远程文件包含，可以直接进行 getshell

4.版本信息

通过对 webshell 的动静态免杀绕过防护软件，进行权限维持。通过修改 webshell 时间戳，放到不被管理员关注的一些深层目录中，去除敏感 shell 函数特征，通过对 shell 流量双向加密去避开常规 waf 检测

webshell 隐藏：通过工具将 webshell 的文件属性进行改变

CVE-2017-0144（永恒之蓝）

利用 Windows 系统的 SMB 协议漏洞来获取了系统的最高权限

SMB 是一个协议服务器信息块，它是一种客户机/服务器、请求/响应协议，通过 SMB 协议可以在计算机间共享文件、打印机、命名管道等资源，电脑上的网上邻居就是靠 SMB 实现的；SMB 协议工作在应用层和会话层，可以用在 TCP/IP 协议之上，SMB 使用 TCP139 端口和 TCP445 端口。

永恒之蓝是在 Windows 的 SMB 服务处理 SMB v1 请求时发生的漏洞，这个漏洞导致攻击者在目标系统上可以执行任意代码。通过永恒之蓝漏洞会扫描开放 445 文件共享端口的 Windows 机器，无需用户任何操作，只要开机上网，不法分子就能在电脑和服务器中植入勒索软件、远程控制木马、虚拟货币挖矿机等恶意程序。

输出到 href 的 xss 漏洞怎么防御

用户的输入放在了 a 标签的 href 属性里面，于是可以用 js 协议来执行我们构造的 js 代码：javascript:alert(“hello!!!”)

防御：因为 href 属性中多半放的是 url 所以可以规定数据开头必须是 http 或者 https，再对这里进行 htmlspecialchars（,ENT_QUOTES）函数处理。

samesite防御CSRF

samesite是cookie的一个属性，能够确保携带cookie发起请求的网站和请求目标的服务是同站的。也就是当用户访问了一个网站得到的cookie会带上samesite属性，当他访问另一个恶意网站时黑客得到的cookie是另一个samesite属性所以无法通过这个cookie去对用户访问的网站发起请求

CSRF的防御

1.只允许GET请求检索数据

2.检查referer字段保证请求来自同源

3.samesite cookie

4.CSRF Token：用户登录后，生成随机值csrf_token，用户提交的操作类（POST）请求中，提交的表单中携带csrf_token，服务器端判断csrf_token是否正确

5.双重Cookie验证：Cookie中保存csrf_token，用户提交表单中也携带csrf_token，服务器端判断两个值是否一致。
